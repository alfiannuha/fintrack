Project Plan & Technical Documentation
Next.js + Go + MongoDB | Vercel Deploy
 

Versi 1.0  Ģ  2025


1. Gambaran Umum Proyek
FinTrack adalah aplikasi Progressive Web App (PWA) untuk pencatatan keuangan personal yang dapat diinstall di homescreen smartphone. Aplikasi ini dirancang untuk pengguna individu maupun pasangan/keluarga yang ingin mengelola keuangan bersama melalui sistem wallet berbagi (shared wallet).

1.1 Tujuan Aplikasi
Ģ Mencatat semua pemasukan dan pengeluaran dengan cepat (target input < 5 detik)
Ģ Mengetahui uang lari ke mana melalui kategorisasi otomatis
Ģ Mengontrol budget bulanan per kategori dengan indikator visual
Ģ Melihat tren keuangan pribadi melalui grafik dan laporan bulanan
Ģ Bukan untuk akuntansi formal - tidak ada jurnal debit/kredit

3. Desain Database (MongoDB)
3.1 Konsep Shared Wallet
Semua transaksi, budget, dan kategori terikat ke wallet_id, bukan ke user_id. Dua user dengan kode wallet yang sama akan melihat data yang identik - cocok untuk keuangan bersama (pasangan, keluarga).
Contoh: User A dan User B memiliki kode  ABCDEF  ? semua transaksi, budget, dan kategori tersedia di kedua akun secara real-time.

3.2 Collection: users

FieldTipeKeterangan_idObjectIdPrimary key, auto-generated oleh MongoDBwallet_idObjectIdForeign key ke collection walletsnamestringNama lengkap useremailstringEmail unik, digunakan untuk loginpassword_hashstringBcrypt hash dari passwordcreated_attimestampWaktu registrasi

3.4 Collection: transactions

4. Alur Autentikasi & Onboarding
4.1 Tiga Jalur Masuk
Aplikasi mendukung tiga skenario onboarding:

1. Registrasi baru - user belum punya kode wallet
? Isi nama, email, password
? Sistem generate wallet code unik 6 karakter (contoh: ABCDEF)
? Kode ditampilkan dan bisa dibagikan ke orang lain
? User langsung masuk ke dashboard

2. Join via kode undangan - punya kode dari teman/pasangan
? Input kode 6 karakter
? Sistem validasi kode dan tampilkan nama wallet
? Isi nama dan email (autentikasi via magic link/OTP)
? Bergabung ke wallet yang ada, data langsung tersinkron

3. Login existing user
? Input email + password
? Backend return JWT access token (15 menit) + refresh token (30 hari)
? Refresh token disimpan di HttpOnly cookie untuk keamanan
? Redirect ke dashboard

4.2 Catatan Teknis Auth
Ģ Setiap request ke protected endpoint harus menyertakan Bearer token di header Authorization
Ģ Middleware Go inject wallet_id ke context dari JWT payload
Ģ Wallet code harus unik - gunakan collision check saat generate, simpan dengan unique index di MongoDB
Ģ Untuk join via kode, gunakan magic link email atau OTP 6 digit untuk verifikasi identitas

5. Fitur Aplikasi
5.1 Quick Transaction (Core Priority)
Fitur utama aplikasi. Target: user bisa mencatat transaksi dalam kurang dari 5 detik.

Ģ Input amount dengan numpad besar, mudah dioperasikan dengan satu tangan
Ģ Toggle cepat antara Income dan Expense
Ģ Category picker berupa grid icon, tidak perlu scroll panjang
Ģ Date default hari ini, bisa diubah
Ģ Note bersifat opsional, bisa dilewati
Ģ Submit langsung memperbarui dashboard secara real-time

5.2 Kategori Default
Expense (pengeluaran):
Ģ Makan, Transport, Tagihan, Belanja, Hiburan, Kesehatan, Lainnya
Income (pemasukan):
Ģ Gaji, Freelance, Bonus, Investasi, Lainnya
Pengguna dapat menambah, mengubah, dan menghapus kategori kustom. Kategori kustom terikat ke wallet sehingga semua anggota wallet melihat kategori yang sama.

5.3 Dashboard Keuangan
Dashboard menjawab 3 pertanyaan utama pengguna:
Ģ Berapa uang tersisa bulan ini? ? Card saldo dengan warna hijau/merah
Ģ Ke mana pengeluaran terbesar? ? Pie chart breakdown kategori
Ģ Bagaimana tren pengeluaran? ? Grafik garis harian
Komponen dashboard:
Ģ Total Income bulan ini
Ģ Total Expense bulan ini
Ģ Net Balance (Income - Expense)
Ģ Pie chart pengeluaran per kategori
Ģ Grafik pengeluaran per hari (line chart atau bar chart)
Ģ List budget dengan progress bar per kategori

5.4 Budget Control
User dapat menetapkan batas pengeluaran per kategori per bulan. Sistem akan:
Ģ Menampilkan progress bar: "Rp 900.000 / Rp 1.500.000 - 60% used"
Ģ Menampilkan warning kuning saat penggunaan mencapai 80%
Ģ Menampilkan alert merah dan notifikasi push saat budget habis
Ģ Budget bersifat per-wallet, semua anggota melihat budget yang sama

5.5 Riwayat Transaksi
Tampilan list transaksi dengan fitur filter:
Ģ Filter berdasarkan rentang tanggal
Ģ Filter berdasarkan kategori (multi-select)
Ģ Filter berdasarkan tipe: Income / Expense / Semua
Ģ Transaksi dikelompokkan per tanggal (grouped by date)
Ģ Setiap transaksi bisa diedit atau dihapus
Ģ Pull-to-refresh untuk memuat data terbaru

5.6 Monthly Report
Laporan ringkasan per bulan yang menampilkan:
Ģ Total Income, Expense, dan Saving bulan tersebut
Ģ Perbandingan dengan bulan sebelumnya (naik/turun dalam persen)
Ģ Breakdown per kategori dengan persentase
Ģ Top 3 kategori pengeluaran terbesar
Ģ Tombol export ke PDF atau CSV

5.7 Recurring Transaction
Untuk transaksi rutin bulanan seperti gaji, sewa, tagihan internet:
Ģ User setup rule: kategori, jumlah, tanggal tiap bulan, catatan
Ģ Cron job di backend Go berjalan setiap hari untuk memeriksa dan menambah transaksi
Ģ User bisa mengaktifkan/menonaktifkan rule tanpa menghapusnya
Ģ Riwayat semua transaksi yang dibuat dari recurring dapat dilihat terpisah

5.8 Fitur Lanjutan (Phase 3)
Auto Categorization:
Ģ Mapping keyword ? kategori: "McDonalds" ? Makan, "Shell" ? Transport, "PLN" ? Tagihan
Ģ User bisa menambah dan mengedit mapping rules
Ģ Kategorisasi otomatis saat note diisi, user tetap bisa mengubah sebelum submit

Financial Insight:
Ģ "Pengeluaran makan naik 30% dibanding bulan lalu"
Ģ Deteksi anomali: pengeluaran jauh di atas rata-rata harian
Ģ Rekomendasi kategori yang bisa dihemat berdasarkan tren

OCR Receipt:
Ģ User foto struk belanja menggunakan kamera
Ģ Integrasi Google Cloud Vision API atau Tesseract.js
Ģ Sistem extract jumlah, merchant, tanggal dari struk
Ģ User konfirmasi dan edit data sebelum menyimpan transaksi

7. Konfigurasi PWA
7.1 Installable di Homescreen
Agar aplikasi bisa diinstall di homescreen Android dan iOS, diperlukan:
Ģ File manifest.json dengan name, short_name, icons (192x192 dan 512x512), start_url, display: standalone
Ģ Paket next-pwa atau @ducanh2912/next-pwa untuk integrasi Next.js
Ģ HTTPS wajib untuk service worker (Vercel sudah menyediakan secara default)
Ģ iOS Safari memerlukan meta tag apple-mobile-web-app-capable di _document.tsx

7.2 Offline Mode
Strategi caching menggunakan service worker:
Ģ Cache-first untuk aset statis (JS, CSS, gambar)
Ģ Network-first untuk data API dashboard dan transaksi
Ģ Background sync: transaksi yang diinput saat offline akan otomatis dikirim saat koneksi pulih
Ģ IndexedDB sebagai penyimpanan lokal untuk antrian transaksi offline

7.3 Push Notification
Notifikasi push menggunakan Web Push API:
Ģ Notifikasi saat budget mendekati limit (80% dan 100%)
Ģ Reminder recurring transaction yang akan tiba (H-1)
Ģ Opt-in per user, bisa dimatikan dari pengaturan
Ģ Backend Go mengirim notifikasi via VAPID keys ke browser

8. Strategi Deployment
8.1 Frontend (Vercel)
Ģ Push ke branch main di GitHub otomatis trigger deploy ke Vercel
Ģ Environment variables: NEXT_PUBLIC_API_URL, NEXT_PUBLIC_VAPID_KEY
Ģ Preview deployment otomatis untuk setiap pull request
Ģ Domain custom dapat dikonfigurasi di Vercel dashboard

8.2 Backend Go (Railway / Fly.io)
Ģ Dockerfile multi-stage untuk binary Go yang kecil (< 20MB)
Ģ Environment variables: MONGO_URI, JWT_SECRET, VAPID_PRIVATE_KEY, CORS_ORIGIN
Ģ Health check endpoint: GET /health untuk monitoring
Ģ Railway menyediakan managed MongoDB opsional, atau gunakan MongoDB Atlas (disarankan)

8.3 Database (MongoDB Atlas)
Ģ Gunakan cluster M0 (free tier) untuk development, M2/M5 untuk production
Ģ Index yang diperlukan: wallet_id + date pada collection transactions, code pada wallets
Ģ Aktifkan MongoDB Atlas Backup untuk production
Ģ Connection string disimpan sebagai environment variable, tidak di-commit ke repository

10. Catatan Teknis Penting
Ģ Gunakan int64 (bukan float) untuk menyimpan jumlah Rupiah guna menghindari floating point error. Tampilkan dengan format Intl.NumberFormat("id-ID", { style: "currency", currency: "IDR" })
Ģ Semua query ke MongoDB harus selalu menyertakan wallet_id sebagai filter utama untuk memastikan isolasi data antar wallet
Ģ Kode invitation 6 karakter sebaiknya menggunakan karakter yang tidak ambigu: hindari 0/O, 1/I/l. Gunakan charset: ABCDEFGHJKLMNPQRSTUVWXYZ23456789
Ģ Untuk iOS PWA, service worker push notification memiliki keterbatasan. Fitur notifikasi push mungkin hanya berfungsi penuh di Android
Ģ Rate limit endpoint auth (register, login, join) untuk mencegah brute force: maksimal 10 request per IP per menit
Ģ Swagger docs dapat diakses di /swagger/index.html pada backend. Pastikan endpoint ini hanya aktif di environment development
